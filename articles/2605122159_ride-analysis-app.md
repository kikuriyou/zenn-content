## はじめに

最近よく自転車に乗るのですが、ランニングやライドを記録するアプリとしてStravaが多くのサイクリストに利用されています。無料版でもスマートウォッチやサイコン、スマートローラーなどと自動的に連携してくれるので、筆者もライド記録を保存したり見返すために愛用しています。これでも十分便利なのですが、半ば知識が付いてくるともう少しリッチな分析をしたり計画を立てたくなってきます。Stravaにも分析機能はあるようなのですが、課金ユーザー向け機能であるため利用には二の足を踏んでいたりします。

そんな中、[Strava API](https://developers.strava.com/)が提供されていることに気づき、これを使えば自分で分析機能を作れるなと思い、実際に作ってみました。

自分がほしいのは、具体的には以下のような機能です。

- ライド・トレーニング管理機能
  - ライド記録を振り返る...①
  - 運動強度ごとのパワー分布や心拍数分布を可視化する...②
  - 運動強度ごとに色分けされたパワーや心拍数の推移をチェックする...③
  - 直近のトレーニング負荷や疲労度などのコンディションを可視化する...④
- ライド・トレーニング計画機能
  - 目標のレースやイベントに向けて運動強度別・目的別のトレーニング計画を立てる...⑤
  - 直近のコンディション(トレーニング負荷や疲労度)にもとづいて次のワークアウトを提示する...⑥

これらを実現するためにAntigravityを使ってアプリを開発したので、その内容をまとめていきます。振り返りや可視化のための分析アプリ本体と、直近のコンディションを元に次のワークアウトをサジェストするエージェント機能を作ったので、これらのうち本記事では分析アプリ本体について説明します。エージェント機能は別途まとめる予定です。

## 作ったアプリ「PerfRide」の概要

作ったのはStrava APIを使用してロードバイクのライド記録・トレーニング計画を管理するWebアプリです。「PerfRide」と呼んでいます。

https://github.com/kikuriyou/PerfRide

Experimentalな機能も含めて、主に以下を実装しています。

| 機能              | 内容                                       | 対応する機能 | Strava連携 | Experimental |
| ---------------- | ------------------------------------------ | ----------------- | :--------: | :----------: |
| Dashboard        | アクティビティ表示、Fitness/Fatigue/Formの表示 | ①, ②, ③, ④      |    必要    |              |
| Weekly Plan      | 今週のワークアウト計画の表示                   | ⑤, ⑥            |  一部必要  |              |
| Climb Simulator  | パワー・体重からあるコースのタイムを物理演算で計算 | —               |  一部必要  |      ✓      |
| Pace Optimizer   | コースプロファイルに基づく最適ペース配分計算       | —              |  一部必要  |      ✓      |
| Training Planner | 目標レース日から逆算してトレーニングプラン生成     | ⑤              |    不要    |      ✓      |
| Settings         | FTP・体重・最大心拍などのユーザーパラメータ設定     | —              |    不要    |              |

### Dashboard

Stravaと連携すると、直近のライド一覧と週間サマリー（走行回数・距離・獲得標高・走行時間）が表示されます。また、過去約13週間分のアクティビティデータからフィットネス指標を算出し、以下の3つの指標と距離や獲得標高の推移をチャートで可視化します。

- **Fitness（CTL: Chronic Training Load）**: 長期的なトレーニング負荷の蓄積。高いほど体力がついている状態
- **Fatigue（ATL: Acute Training Load）**: 直近のトレーニングによる疲労度。高いほど疲れが溜まっている状態
- **Form（TSB: Training Stress Balance）**: Fitness − Fatigue。コンディションの良し悪しを示す（+10〜+25がレースに最適）

各ライドをクリックすると、心拍ゾーン分布・パワープロファイル・標高チャートなどの詳細を確認できます。

![](/images/articles/ride-analysis-app/dashboard.png)
*ダッシュボードページ*

### Weekly Plan（今週のワークアウト計画）

「今週どんなトレーニングをすればよいか」を曜日ごとに並べて表示する機能です。Dashboardには本日のセッションだけを抜粋したカードを置き、週間プランページでは1週間分のセッション一覧（種別・目標TSS・状態）と過去のプラン履歴を確認できます。

![](/images/articles/ride-analysis-app/weekly_plan.png)
*週間プランページ*

生成自体は別途用意したエージェントが担当し、毎週月曜の早朝に翌週分を自動で組み立てます。エージェントが直近のFitness/Fatigue/Formやユーザー設定をもとに期分け（Base→Build→Peak→Taper）に沿ったセッションを提案する仕組みです。詳細は別記事で扱います。


### Experimental機能について

これ以降で紹介する Climb Simulator / Pace Optimizer / Training Planner の3つは、現時点では Experimental扱いとしています。まだ試行錯誤中で、文献などを参考に実装しているものの導線やパラメータなど細かい部分を整理しきれていないこと、また AntigravityやLLMのサポートで気軽に作れた反面、とりあえず作ってみた状態であることが理由です。

Climb Simulatorは見込みタイムを計算する機能で、パワー、体重、距離、獲得標高を入力すると、物理演算に基づいて予測タイムを計算します。Stravaに接続していれば、お気に入り登録しているセグメントでの予測タイムを計算できます。

![](/images/articles/ride-analysis-app/simulator.png)
*Climb Simulatorページ*

Pace Optimizerはペース最適化の機能で、コースプロファイルにもとづいて最適なペース配分を計算します。この機能は、2025年に発表された論文 ["A numerical design methodology for optimal pacing strategy in the individual time trial discipline of cycling"](https://link.springer.com/article/10.1007/s12283-025-00493-9) をベースに実装しています。論文の主張である「登りでは高パワー、下りでは低パワーにすることで、Normalized Power（NP）を一定に保ちながらタイムを短縮できる」という戦略にもとづいています。速度が低い登りでは空気抵抗が小さいため、追加のパワーがより効率的にタイムに反映される、という考え方のようです。まだ改善の余地はありそうですが、最新の研究や文献を取り入れた機能です。

![](/images/articles/ride-analysis-app/optimizer.png)
*Pace Optimizerページ*

Training Plannerは、目標レース日を設定して残り期間から逆算してフェーズごとのトレーニングプランを自動生成します。おおよそ以下の配分で負荷ごとのフェーズが決まるようになっています。

| フェーズ | 期間比率 | 目的                   |
| -------- | :------: | ---------------------- |
| Base     |   35%    | 有酸素基盤の構築       |
| Build 1  |   25%    | テンポ・閾値の向上     |
| Build 2  |   20%    | VO2max・高強度対応     |
| Peak     |   10%    | レースシミュレーション |
| Taper    |   残り   | 疲労回復・調整         |

![](/images/articles/ride-analysis-app/planner.png)
*Training Plannerページ*


## アーキテクチャ

### システム構成の概要

PerfRideのアプリ部分は「フロントエンド + BFF + 外部API」を基本にした構成で、追加で次ワークアウトのサジェストや補正を担うエージェント機能を切り出して用意しています。

フロントから直接ではなくNext.jsのAPI Routesを介してStrava APIを呼び出すBFFパターンを採用することで、アクセストークンの保管・リフレッシュ・キャッシュ層をサーバ側に持たせています。これによりブラウザにシークレットを露出させず、レート制限対策のキャッシュも一箇所で済みます。

ワークアウト履歴を保持するDBは持たず、Stravaから取得されるデータを正として扱っています。アクティビティ一覧・詳細・セグメント情報はその都度Strava APIから取得し、API Routes（BFF層）で一定時間キャッシュすることで、アプリ側にデータを抱えないまま、同期処理を不要にしつつレート制限対策も兼ねています。アプリ側で永続化するのはユーザーごとの設定・トークン・週次プランなどに限られ、これらはCloud Storageに保存しています。スキーマ変更や環境ごとの運用が不要で、小規模利用であればこの構成で十分と考えています。

次ワークアウトのサジェスト・補正のエージェント機能のような、「LLM＋ドメインロジック」が必要な処理はPythonの方がライブラリが充実している(+筆者が慣れている)ため、Cloud Run上の独立したFastAPIサービスに切り出しています。この詳細は別記事で扱います。

### Strava APIのレート制限への対応

Stravaの[公式ドキュメント](https://developers.strava.com/docs/rate-limits/)によると、2026年5月時点でデフォルトのレート制限は以下の通りで、適用単位はアプリケーションごとで上限を超えると `429 Too Many Requests` が返るようになっています。

| 区分                  | 15分     | 1日       |
| --------------------- | :------: | :-------: |
| Overall（全体）        | 200req   | 2,000req  |
| Non-Upload（読み取り中心）| 100req   | 1,000req  |

リクエスト数のレート制限はあまり問題にならないのですが、影響が大きいのが **Connected Athlete = 1** という制限です。この制限によってアプリに認可できるユーザーが累計で1人までになるので、自分以外のユーザーに試してもらうことができません。Connected Athleteを増やすには Developer Program のフォーム提出が必要で、申請すると1, 2日ほどで上限を上げてくれました。


## 開発にあたって

構築にあたっては、フロントエンドとBFF（Next.jsのAPI Routes）をTypeScriptで、LLMやドメインロジックを伴うエージェント部分をPython（FastAPI）で、という役割分担に落ち着いています。筆者は元々業務ではPythonを主に使っていてTypeScriptには精通していないのですが、Web側をTypeScriptで書くこと自体は今回あまり問題にはなりませんでした。その理由として、(このアプリが小規模であることは前提として)主に以下の2点が挙げられます。

1. **LLMとの会話でロジックを把握できる**  
   「このチャートのロジックを説明して」「実装箇所を教えて」と会話することで、ロジックの概要を把握したりコードの関連箇所を読んで処理内容を理解できます。
2. **物理定数や数式の簡単な変更は言語の影響が少ない**  
   `const GRAVITY = 9.81` のような定数や、基本的な四則演算・三角関数を使った数式の変更は、該当する部分さえわかれば言語に左右されずに編集できます。ただし、最適化のような繰り返し計算などは実装自体を理解する必要が出てくるでしょう。

一方で、エージェント部分（次ワークアウトのサジェスト・補正、別記事で説明）はLLM周辺ライブラリの充実度や筆者自身の慣れの度合いでPythonに分があったため、Web側と切り離して用意しています。試行錯誤の1つとしてWebのバックエンドにPythonを使うことも検討しましたが、フロントエンドとの結合部分でデバッグが増えた印象があり、結果として「UIに近い層はTypeScript、LLM・推論を抱える層はPython」という分担に収まっています。ただ、これはIF定義を丁寧に記述すれば問題はなくなるかもしれません。いずれにせよ、これらは開発するアプリやシステムの規模、扱うロジックの種類にもよるので、今後も良い判断方針を探っていきたいところです。

## まとめ

本記事では、Strava APIを利用してシンプルな構成で作成したPerfRideを紹介しました。AntigravityをはじめとするAgentic Codingツールを使うと、専門外の言語やライブラリでも自分のニーズに合った分析機能を気軽に短時間で作ることができます。結果として有料機能のサブスク費用の節約につながることもあるので、個人用途ではかなりおすすめできるアプローチかと思います。みなさんもぜひお試しください。
