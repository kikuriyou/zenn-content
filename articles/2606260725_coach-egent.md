## TL;DR

ロードバイク用のライド分析アプリ PerfRide のコーチエージェントを主に 3 つのアプローチで設計改善しました。

1. **コンテキスト最適化**  
グラウンディングを、自由度の高い Web 検索から、トレーニング目標と専門知識に絞る。

2. **伝える内容の透明性**  
理由と根拠・確信度・リファレンスを伝える。確信度を自然な表現で伝える。

3. **適切なタイミングでの働きかけ**  
アプリの使用パターンをもとに定期実行とトリガーを設定。

結果、信頼できる知識・理由・根拠による納得感により、エージェントの提案を受け入れやすくなり、また、アプリの使用パターンに沿った実行・トリガーで何も考えずに毎日のプランに取り組めるようになりました。

https://github.com/kikuriyou/PerfRide


## PerfRide にコーチエージェントを入れる

PerfRide は Strava API と連携して、CTL / ATL / TSB といったフィットネス指標やパワーゾーン分布を可視化するライド分析アプリです。アプリの詳細は[以前書いた記事](https://zenn.dev/kikuriyou/articles/2605122159_ride-analysis-app)を参照ください。
このアプリを使う中で、アクティビティの連続日数を伸ばそうと無理したり、逆に調子が良いため高強度トレーニングを連日行った結果、体調を崩すことが何度かあり、休息を計画に織り込んだり効果を意識したメニューを取り入れる必要性を感じていました。そこで、自分のアクティビティをもとにおすすめメニューを提案・計画してくれるコーチエージェントを作ることにしました。

:::message
実装した内容は、実際のコーチに相談する前のセルフトレーニングの質を上げることをスコープにしています。医療・専門的アドバイスの代替を意図したものではありません。
:::

最初のトライアルとしては以下のようなアーキテクチャで構築しました。アプリ本体とは別に立てたコンテナで、Strava API から取得したアクティビティ情報を ADK, Gemini 3 Flash に渡し、Web 検索によるグラウンディングで事実ベースの回答にして信頼性向上を狙っています。

![](/images/articles/coach-egent/architecture_1.png)
*アーキテクチャ(改善前)*

ただ、実際には検証のために同じ入力を与えた場合でも、おすすめ内容が安定しない、例えば表示ごとに真逆のおすすめをする（ある時は低強度、ある時は高強度をおすすめする）ことがあり、結果、信頼性は十分と言えない表示内容になっていました。


## エージェントの設計改善のための調査
ここで、エージェントの設計改善のための調査を公開情報ベースで行っています。各チャットサービスの Deep Research を利用して取得したリソースの中から、特に重要と思われるものに絞って個別に深掘りしています。主要なリソースはこちらの [NotebookLM](https://notebooklm.google.com/notebook/85c5da2f-79d4-4588-a4c3-22753d5ca10f) にまとめているので、興味があれば参照ください。

設計改善のアプローチとしてはいくつもあるのですが、今回の文脈で効果が大きいと考えられる以下の3つに絞って設計に組み込み、実装しています。

- ① コンテキスト最適化
- ② 伝える内容の透明性
- ③ 適切なタイミングでの働きかけ

以降でそれぞれの詳細を説明します。

## ① コンテキスト最適化

Web 検索のグラウンディングを行っても、巷のプラクティスと若干異なる説明が返ることが少なくありませんでした。取得したページを確認すると、原因は以下のようなケースが多かったです。

- 多くのページを参照するため、意図の異なる情報が混ざる
- 非専門家の記事や、微妙に領域の違う記事（ランニングや筋トレ）が混ざる

これは、Web ページが玉石混交である現状を考慮すると一定仕方ないことかと思います。そこで、情報源を以下のように絞るように変更しました。

- 事前に調査、整理して格納した専門知識
  - power_zones.md: ゾーンごとのパワーモデル、ゾーンごとの FTP
  - periodization.md: トレーニングのピリオダイゼーション など
- 該当する内容が見つけられない場合、Fallback として Web 検索する
  - 検索対象のドメインは事前に指定したものに限る ("trainingpeaks.com", "mywhoosh.com", "forum.intervals.icu" など)

この変更によって、疲労度やトレーニング効果を考慮してメリハリのあるプランを出力したり、直前アクティビティの振り返りやおすすめプランの理由も明瞭になり、複数回の生成においても再現性のある結果が得られるように改善されています。


## ② 伝える内容の透明性

コーチエージェントを実装した当初は、「今日はスイートスポットインターバルで持久力を上げましょう」などというように、単純に「xxをしましょう」というおすすめが多かったのですが、単におすすめを言われたとしても「一般論としてはそうだよね」「データのどこを見てそう思った？」という気持ちが湧いてきます。この問題に関しては、以下のように理由や根拠、確信度、リファレンスを出力させることで透明性を上げることで対処しています。

- 理由とともに高強度をおすすめする
  - 「疲労度が x 未満であるため...」というデータの参照箇所が読み取れるようになる
- データだけでは曖昧な場合でも表現の変化を通して確信度を伝えることでおすすめの確からしさを汲み取ることができる
  - 直近 X 日のデータから(=確信度:高)
  - データから推測すると(=確信度:中)
  - …かもしれません(=確信度:低)
- リファレンスの情報源を常に表示する
  - 納得できない場合でもその根拠を辿ることができるようになる
  
合わせて、プロンプトで判断基準も調整しています。例えば、疲労度合いを評価する時、「コンディションを示す TSB の範囲だけで判断せず、直近 2〜3 日のライドを強度で分類してから決める」「『低強度が続いている』等の断定は直近 3 日の強度を確認してから使う」などといった、データの具体的な確認方法の指示を加えています。
このように変更した結果、明らかに間違ったおすすめは抑えられることを確認しています。また、理由がわかるとおすすめが多少間違っていても許せるようになる体感の変化も感じられました。


## ③ 適切なタイミングでの働きかけ

コーチエージェントの動作タイミングとして、元々は毎日の初回閲覧時におすすめメニューを更新するようなシンプルな設定としていました。しかしこの方法では、いざトレーニングを行った後、次のおすすめを確認したくなった時に「コメントないの？」という微妙な気持ちにさせられていました。

![](/images/articles/coach-egent/trigger_1.png)
*コーチエージェントの動作タイミング(改善前)*

この問題については、まずはこうあってほしいというアプリの使用サイクルを見直すことから始めており、

- 毎週月曜に週間プラン作成
- 毎日のトレーニング後に、最新状況にもとづいて翌日のプランを更新
  - トレーニング時に使用しているワークアウトアプリ上でもプランを登録
- 最新状況から疲労蓄積が懸念される場合はアラート通知

というサイクルを定義をしました。

![](/images/articles/coach-egent/trigger_2.png)
*コーチエージェントの動作タイミング(改善後)*


この変更は定期実行とイベント駆動で実装されており、それ自体は使い古された方法ではあるものの、利用サイクルに沿った必要な動作を行えるため、ユーザーは何も考えずにエージェントに従うことで毎日のプランに取り組めるようになります。


## 変更後のアーキテクチャと効果

ここまでに説明した内容をまとめて適用したアーキテクチャは以下のようになります。ADK, Gemini から各情報源にアクセスする骨子は変わりませんが、

- コンテキスト最適化のためのグラウンディングの参照先が変わり、
- 伝える内容の透明性を改善するために ADK や Gemini に入力するプロンプトを調整し、
- 適切なタイミングでの働きかけのための定期実行を実現するために Cloud Scheduler を追加、合わせて入力情報（カレンダー情報、天気予報）や出力動作（ワークアウトアプリへのプラン登録）も追加する、

といった変化があるかと思います。

![](/images/articles/coach-egent/architecture_2.png)
*コーチエージェントのアーキテクチャ(改善後)*


この変更により、以下のような効果が得られています。

- 信頼できる知識・理由・根拠による納得感により、エージェントの提案を受け入れやすくなる。例えば、休息の罪悪感が抑えられたり、高強度メニューの実施判断がスムーズに行える。
- アプリの使用パターンに沿った実行・トリガーで何も考えずに毎日のプランに取り組める。


## おわりに

本記事では、ライド分析アプリへのコーチエージェント機能を通して、エージェントの設計改善の試行錯誤について説明しました。このアプリでは、コンテキスト最適化、伝える内容の透明性、適切なタイミングでの働きかけの3つによる効果が大きかったですが、アプローチとしては他にも色々ありますので、ぜひ [NotebookLM](https://notebooklm.google.com/notebook/85c5da2f-79d4-4588-a4c3-22753d5ca10f) も参照して、ご自身の事例に役立つ知見を探してもらえればと思います。本記事が少しでもみなさまのエージェント開発の役に立てば幸いです。
